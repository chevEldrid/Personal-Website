Introduction to OpenGL

March 9, 2018

It's crazy to think I'm already almost done with the second week of classes, but time really does fly here. Hopefully I remain this on top of my assignments for the rest of this semester...

OpenGL

For a long time I've wanted to be a game designer. Taking amazing stories and building them into experiences for a wide audience to enjoy seems like a dream come true. Multiple times I would get together with some of my closest friends and draft up complicated ideas and mechanics that seemed no farther than an intense afternoon away. Yet, there was always a hurdle that stunted progress past the mid-eighties. The reason why I've primarily stuck with designing in text windows, or "Text-Adventures," has been my complete and total fear of graphics. Where do they come from? How do you make them? What sorcery allows sprites to jump or characters in FPS-style games to aim down sights? Sure, I'm familiar with the ideas of Sprite sheets or 3D modelling, but only at a very high level. Finding I learn best in a structured environment this illogical fear of pictures is part of what made me take Computer Graphics here at the University of Auckland. I wanted to know how these things work at a base level, how to understand terminology with CG and eventually, how to incorporate graphics into experiences in games and software that extend past white text on a black screen.

Enter, OpenGL. Introduced to us on the second day of class OpenGL is "...a cross-language, cross-platform application programming interface (API) for rendering 2D and 3D vector graphics." (Wikipedia) The first building block of many modern graphic design programs. While big designers might not go down to the OpenGL level over say, Autodesk Maya, it's still incredibly helpful to understand on what base programs like Maya are being run.

Visual Studio

The way we would be using OpenGL is through a C++ library in an online tool with instructions provided to us on how to install the setup at home if you wished. Knowing how often I'm going to want to tinker with design and just mess around I knew setting up a home environment was a must, and the easiest way appeared to be through Visual Studio. A little background: one of the first languages I was introduced to when I started getting into programming was C++. I haven't used it quite as much as Java but through classes like embedded design it hasn't strayed too far from my mind. I was used to Netbeans mostly but thought, foolishly, how hard can Visual Studio be to learn? The answer in my case was very. Initial installation wasn't so much a problem, nor was setting up my first program. So for using base C++ functionality Visual Studio is pretty great, it takes almost no time to create and test .exe files or programs with constant feedback from the software. However when it comes to adding external libraries, sometimes Visual is a nightmare. Since almost every installation has a different setup, knowing where to put library and header files in the swamp of directories is a nightmare. Most of the advice online is outdated by the time you find it and then digging around yourself if you don't know exactly what you're looking for can be like feeling your way through a labrynth. Eventually, I had given up on actually putting OpenGL files directly onto my computer and instead using the built-in package installer for Visual. This is a good solution, but does require installing the packages with every project. It wasn't until another student in my class pointed out exaclty where to store the files in Visual Studio Community Edition 2017 that I got libraries working correctly with no package installation. It was a lot of work creating my first circle.

That being said, since the initial installation using OpenGL has worked wonderfully and testing homework problems by spinning them up or learning the differences between a triangle fan and triangle strip has been a relatively great time!

This is the shape created by the code above. In the most simplistic terms, your openGL program will consist of three parts: main, init, and display. The main function does what you would expect it to do and after initializing your program, displays the graphic you've created. The window around the graphic is defined in the init() function identifying the overall size of the window while the display() function does the actual graphic definition.

Looking closesly at the code pictured above you will see two key method calls: glBegin(), and glEnd(). Inbetween these two I've called a very simplistic helper method that basically places vertices from a list above onto the window created by init(). Then, depending on the argument in glBegin, these verteces will be connected into some kind of shape which here is a "Triangle Fan"; using the initial vertex, trangles are traced with every subsequent pair of verteces back to the original vertex. It's really cool! Even if it takes a minute to wrap your head around.
